B2B 주문 시스템 설계: 최종안

전용몰과 오픈마켓, 두 가지 모델에 최적화된 주문 시스템을 제안합니다.

플랫폼 유형별 주문 시스템 비교

우리의 비즈니스 모델에 따라 고객에게 최적의 경험을 제공하기 위해 주문 시스템을 이원화하여 설계합니다.

🛍️ 전용몰 (단일 공급사)

핵심 목표: 신속성. 단골 고객이 가장 빠르게 주문 완료.

📋

발주리스트

→
🧾

주문서

  • 단순한 2단계 주문: 빠르고 직관적인 `발주리스트 → 주문서` 흐름을 유지합니다.
  • 통합 배송비: 전체 주문 금액 기준으로 배송비를 한 번만 계산하여 안내합니다.
  • 합배송 기능 적용: 예약상품과 일반상품을 묶어 배송비를 절약하는 스마트 기능을 제공합니다.

🌐 오픈마켓 (다수 공급사)

핵심 목표: 유연성. 여러 공급사 주문을 자유롭게 조합.

📋

발주리스트

→

주문서

  • '장바구니' 도입: 주문서로 넘어가기 전, 공급사별로 주문 내역과 배송비를 명확히 확인하는 중간 단계를 제공합니다.
  • 발주리스트 중심: 모든 주문은 발주리스트에서 시작됩니다. (새 상품도 발주리스트에 추가 후 주문)
  • 합배송 기능 동일 적용: 전용몰과 마찬가지로 합배송 기능을 통해 고객 혜택을 강화합니다.

페이지별 필수 항목

📋 발주리스트

  • 합배송 가능 알림 표시
  • 상품 정보 (이름, 공급사, 단가)
  • 수량 입력 칸
  • 선택 체크박스
  • [장바구니/주문하기] 버튼
  • [편집/삭제] 버튼

🛒 장바구니 (오픈마켓)

  • 배송지 표시 및 변경
  • 공급사별 상품 목록
  • 합배송 상품 포함 금액 표시
  • 공급사별 배송비 안내
  • 총 주문금액 (요약)
  • [주문하기] 버튼

🧾 주문서

  • 배송지 정보 (최종 확인)
  • 주문 상품 목록 (확인용)
  • 최종 결제금액
  • 쿠폰 자동 적용 (최대 혜택)
  • 결제 수단 선택 (여신 포함)
  • 세금계산서 등 증빙 신청
  • 품절 시 대체상품 변경 동의
  • [결제하기] 버튼

3. 스마트 배송비

오늘 주문금액에 예약상품 금액이
합산되어 배송비 0원 혜택

개발 브리핑 주요 논의 안건

실제 개발에 착수하기 전, 정책과 세부 로직을 명확히 정의하기 위해 논의가 필요한 사항들입니다.

1. 정책 및 운영

  • 합배송 프로세스: 선결제된 예약상품의 '배송 준비 완료' 상태는 어떻게 추적하고(자동/수동), 고객에게 어느 시점에 알림을 노출할까요?
  • 품절 시 대체상품: '유사 상품' 선정 기준은 무엇이며, 발생 가능한 차액은 어떻게 처리할 계획인가요?
  • 공급사 정책 관리: 오픈마켓의 경우, 공급사가 배송비 정책을 직접 관리하는 어드민 기능이 필요한가요?

2. 사용자 경험(UX)

  • '장바구니' 화면 설계: 오픈마켓에서 공급사별 주문내역과 배송비를 어떻게 명확하고 간결하게 보여줄 수 있을까요?
  • 합배송 경험 시각화: '배송 준비 완료' 알림(배너/팝업)과 배송비 절약 혜택을 어떻게 명확하게 보여줄까요?
  • 여신 한도 초과: 주문을 막을까요, 아니면 다른 결제수단과 복합 결제를 지원할까요?

3. 기술 및 API

  • 합배송 API 설계: '배송 준비 완료된 예약상품' 조회 API, 합배송 금액을 포함하는 배송비 계산 API의 데이터 구조는 어떻게 설계해야 할까요?
  • '장바구니' 페이지 API: 발주리스트에서 선택한 상품 목록을 받아, 공급사별로 그룹핑하고 배송비/쿠폰까지 계산하여 반환하는 API가 필요합니다.
  • 실시간 예외 처리: 주문 중 재고/여신 변동 시점이나 배송 불가 지역 선택 시, 어떻게 검증하고 사용자에게 안내할까요?
